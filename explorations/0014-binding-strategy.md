# Exploration 0014: Binding Strategy — UniFFI vs Hand-Written

Status: Distilled into ADR-0010

## Question

Should the reworked bindings use UniFFI (single definition → Java + Python)
or continue with hand-written wrappers (JNI for Java, PyO3 for Python,
wasm-bindgen for JS/WASM)?

JS/WASM is out of scope for this comparison — wasm-bindgen is the only
viable option there regardless.

## Current State (origin/main)

Each binding is hand-written:

- **Java** — raw JNI via the `jni` crate. 515 lines total.
  Every function manually converts `JString → String`, handles
  null-as-Option, calls the core function, then match-arms every error
  variant into a specific Java exception. The `init` wrapper alone is
  ~80 lines.

- **Python** — PyO3. ~605 lines total. Much lighter: `#[pyfunction]`
  gives direct Rust↔Python type mapping. `init` wrapper is ~30 lines,
  mostly error mapping.

- **JS/WASM** — wasm-bindgen. ~70 lines. Architecturally different
  (browser localStorage backend). Very thin.

### Pain Points

1. **Java JNI is the bottleneck.** Every argument is a manual
   `JString` extraction with null-check boilerplate. Every error
   variant is a manual match arm. Adding a new parameter or error
   variant requires editing 3+ places.

2. **Error mapping is duplicated.** The same `InitError → exception`
   logic exists in both Java and Python, expressed differently.

3. **No structural guarantee.** Nothing enforces that Java and Python
   expose the same functions with the same signatures. Drift is
   caught only by review.

## Option A: UniFFI for Java + Python

UniFFI lets you define the FFI interface once (either in a `.udl` file
or via proc macros) and generates Java, Python, Swift, Kotlin, and Ruby
bindings automatically.

### What changes

**Core library** exposes a concrete facade layer:

```rust
// In sysand-core, or a thin sysand-ffi crate

#[derive(uniffi::Record)]
pub struct InitOptions {
    pub name: String,
    pub publisher: Option<String>,
    pub version: String,
    pub license: Option<String>,
    pub allow_non_spdx: bool,
}

#[derive(uniffi::Enum)]
pub enum ErrorCode {
    VersionInvalid,
    LicenseInvalid,
    IoError,
    // ... flat enum, same as spec/error-model.md
}

#[derive(uniffi::Error)]
pub struct SysandError {
    pub code: ErrorCode,
    pub message: String,
    pub context: Option<String>,
}

#[uniffi::export]
pub fn project_init(path: String, options: InitOptions) -> Result<(), SysandError> {
    // Calls the internal generic do_init with LocalSrcProject
    // Maps InitError<LocalSrcError> → SysandError
}
```

**Generated output** (automatic, no hand-written code):

Java:
```java
// Auto-generated by UniFFI
public class SysandKt {
    public static void projectInit(String path, InitOptions options) throws SysandException { ... }
}
```

Python:
```python
# Auto-generated by UniFFI
def project_init(path: str, options: InitOptions) -> None: ...
```

### Advantages

1. **One definition, two bindings.** Add a parameter to `InitOptions`
   once; Java and Python both get it. No drift possible.

2. **Eliminates all JNI boilerplate.** The ~515 lines of Java binding
   code become ~0 hand-written lines. Error mapping, null handling,
   type conversion — all generated.

3. **Error model alignment.** The spec already defines a flat
   `ErrorCode` enum and `SysandError` struct. UniFFI's error model
   maps directly to this — no per-command exception classes.

4. **Kotlin/Swift for free.** If mobile or native desktop bindings are
   ever needed, they come at zero additional cost.

5. **Structural consistency by construction.** The projection rules
   in spec/projection-rules.md are *enforced* by the tooling rather
   than by discipline.

### Disadvantages

1. **No generics across FFI.** The current `do_init<P: ProjectMut>`
   pattern can't be exposed directly. Need a concrete facade that
   picks the storage backend (filesystem for Java/Python). This is
   actually what we want — the public API takes a path, not a trait
   object — but it means the facade layer is an additional thing to
   maintain.

2. **Namespace mapping isn't automatic.** UniFFI generates flat
   functions (`project_init`) or methods on objects. The
   `client.project().init()` builder-chain style for Java would need
   a thin hand-written Java wrapper on top of the generated code.
   Alternatively, accept `Sysand.projectInit()` as the Java surface.

3. **Async support is limited.** UniFFI has experimental async
   support, but it's not as mature as manual PyO3/JNI async handling.
   For sysand this matters for network operations (lock update, env
   sync, clone). The spec says Java/Python are "sync by default;
   binding layer manages async runtime internally" — which is what
   UniFFI does (blocks on a Tokio runtime internally).

4. **Build complexity.** UniFFI adds a build step (generating
   bindings) and a dependency on the `uniffi` crate + CLI tool.
   Not huge, but non-zero.

5. **Less control over language-specific types.** Python gets
   generated dataclasses rather than hand-tuned PyO3 types. Java
   gets generated POJOs rather than builder-pattern objects. The
   generated code is correct but may feel less "native" to users of
   those languages.

## Option B: Hand-Written (Current Approach, Improved)

Keep JNI + PyO3 + wasm-bindgen, but systematize the pattern.

### What changes

Define a macro or code-generation layer that reduces the per-command
boilerplate. The facade layer still exists (concrete functions that
map `SysandError`), but the binding code is hand-written per language.

**Java binding for init (reworked):**

```rust
#[unsafe(no_mangle)]
pub extern "system" fn Java_com_sensmetry_sysand_Project_init<'local>(
    mut env: JNIEnv<'local>,
    _class: JClass<'local>,
    path: JString<'local>,
    options: JObject<'local>,  // InitOptions Java object
) {
    sysand_jni::run(&mut env, |env| {
        let path = env.get_string_required(&path, "path")?;
        let opts = InitOptions::from_jobject(env, &options)?;
        sysand_core::project::init(path, opts)
    });
    // sysand_jni::run handles SysandError → SysandException conversion
}
```

**Python binding for init (reworked):**

```rust
#[pyfunction]
#[pyo3(signature = (path, /, **options))]
fn init(path: String, options: Option<InitOptions>) -> PyResult<()> {
    let opts = options.unwrap_or_default();
    sysand_core::project::init(path, opts).map_err(into_py_err)
}

// Registered under: m.add_submodule("project")?.add_function(init)?;
```

### Advantages

1. **Full control over language-native feel.** Java gets builders,
   Python gets keyword args, JS gets optional object literals.

2. **No new build dependency.** PyO3 and wasm-bindgen are already
   in the dependency tree.

3. **Flexible error mapping.** Can use Python-native exception
   subclasses per error code if desired.

4. **Namespace structure.** Python submodules
   (`sysand.project.init()`) and Java inner classes
   (`client.project().init()`) can match the spec exactly.

### Disadvantages

1. **JNI remains painful.** Even with helper macros, each function
   needs JString extraction, option object unpacking, and error
   conversion. With ~20 commands × 5–10 parameters each, this is
   significant ongoing maintenance.

2. **Drift risk.** Nothing structural prevents Java from accepting
   a parameter that Python doesn't, or vice versa.

3. **More code to review.** Every command is implemented 3 times
   (JNI, PyO3, wasm-bindgen) with language-specific concerns.

## Option C: Hybrid — UniFFI for Java, PyO3 for Python

A middle ground: use UniFFI only where the pain is worst (Java JNI),
keep PyO3 for Python (since it's already ergonomic), keep wasm-bindgen
for JS.

This loses the "single definition" guarantee between Java and Python,
but eliminates the JNI boilerplate while keeping PyO3's ergonomics.

Not recommended — it adds complexity without the main benefit (single
source of truth).

## Namespace Concern

The spec requires `client.project().init()` for Java and
`sysand.project.init()` for Python. With UniFFI:

- **Python:** UniFFI generates a flat module. We'd add a thin
  `sysand/project/__init__.py` that re-exports from the generated
  module. ~5 lines per namespace. Trivial.

- **Java:** UniFFI generates a class with static methods. We'd add
  a thin `Project` class with instance methods that delegate.
  ~3 lines per method. Or accept static methods
  (`Sysand.projectInit()`) as the Java API — simpler, still
  discoverable, and actually common in Java FFI libraries.

## Recommendation

**UniFFI for Java + Python.** The argument is primarily about Java:
raw JNI is the single biggest source of boilerplate and the highest
maintenance risk. UniFFI eliminates it entirely. Python comes along
for free, and while PyO3 is fine on its own, having a single
definition for both is strictly better for consistency.

The facade layer (concrete functions that convert internal errors to
`SysandError`) is needed regardless of approach — the spec's error
model demands it. With UniFFI, this facade *is* the binding interface.
Without UniFFI, it's an internal layer that JNI and PyO3 both call.
Same code, different consumers.

The namespace concern is real but small — a thin hand-written wrapper
in each language to organize flat functions into the right namespace
structure. This is ~50 lines per language, written once.

## Open Questions

   directly) or in a separate `sysand-ffi` crate?
2. Does the Java namespace wrapper (`client.project().init()` style)
   justify the complexity, or are static methods acceptable?
3. How does UniFFI's async story work for network-bound commands
   like `lock update` and `env sync`?
